## 一、Auto-Memory Extraction（自动记忆提取）

### 机制
每次对话结束后，一个 **forked subagent** 自动分析最新的消息，提取值得记住的内容并保存到记忆文件。

### 核心设计
```
主 agent 回复 → 触发 extractMemories hook
                    ↓
         forked extraction agent 启动
         （工具权限严格限制：只能读写 memory 目录）
                    ↓
         分析最新消息 → 决定保存/更新/删除哪些记忆
                    ↓
         写记忆文件 → 更新 MEMORY.md 索引
```

### 关键设计点
- ** Mutual exclusion**：如果主 agent 自己写了记忆，extraction agent 跳过（不重复）
- **Turn budget 限制**：最多 5 轮，防止验证陷阱消耗 token
- **只读新消息**：只看最新 N 条消息，不重复读历史
- **工具白名单**：只允许 Read/Glob/Grep + 只读的 Bash，禁止 Write 到 memory 目录外

### 借鉴价值
> OpenClaw 可以做"会话后记忆提取"——每次会话压缩时，让 AI 主动判断哪些内容值得从 transcript 升权到 MEMORY.md

## 二、Memory Taxonomy（记忆分类学）

Claude Code 的记忆分为 **4 种类型**：

| 类型 | 内容 | 保存时机 |
|------|------|---------|
| **user** | 用户角色、偏好、背景知识 | 了解用户角色/偏好时 |
| **feedback** | 用户的反馈（喜好/厌恶） | 用户对结果有反应时 |
| **project** | 项目事实和规则 | 发现项目规则时 |
| **patterns** | 常见代码模式 | 发现复用模式时 |

### WHAT NOT TO SAVE（极其重要）
```
不应保存到记忆：
- 代码模式、架构、文件路径（可从当前项目推导）
- Git 历史（git log 是权威来源）
- 调试方案（fix 在代码里，commit message 有上下文）
- CLAUDE.md 已有的内容
- 临时任务细节、进行中的工作
```

### 核心原则
> **记忆的价值在于"意外发现"，不在于记录已知事实**

Claude Code 甚至有一条规则：
> 如果用户要求保存 PR 列表或活动摘要 → 问他们"其中哪些是让你意外的？"，只保存那部分。

### 借鉴价值
> **对我们最有用的规则**：MEMORY.md 里很多内容其实是"项目 fact"，这些应该从代码里实时读取，而不是静态保存。我们的记忆应该聚焦于：用户偏好、反馈、项目独特规则。

## 三、AutoDream（跨会话记忆整合）

### 机制
定期（默认 24 小时 + 5 个新 session）触发一个后台 agent：
1. 扫描所有 session 的记忆文件
2. 识别陈旧记忆（drift detection）
3. 合并重复记忆
4. 删除过时内容

### 记忆漂移检测（Drift Detection）
```typescript
// 使用记忆前，验证它是否还正确
'- Memory records can become stale. Before answering, verify the memory
   is still correct. If it conflicts with current state, trust what you
   observe now — and update or remove the stale memory.'
```

### 借鉴价值
> OpenClaw 的 memory-tier.py 只做归档，不做"整合"。Dream 思路可以增强：让 AI 定期检查 MEMORY.md 里的记忆，识别哪些已经和当前事实不符。

## 四、Session Compact 服务（会话压缩架构）

Claude Code 的 compact 系统非常成熟，包括：
- `autoCompact`：当 token 快用满时自动压缩
- `microCompact`：更轻量的压缩策略
- `prompt.ts`：压缩 prompt 模板

### 借鉴价值
> OpenClaw 的"每日会话压缩"cron job 可以借鉴其分层层级思想：
> - 微压缩（microCompact）：只保留头尾，压缩中间
> - 完整压缩：生成摘要，替换整个 transcript

## 五、Prompt Cache 共享（Subagent 优化）

Claude Code 的 subagent 设计里，所有 fork children 的 **tool 定义完全相同**，只有 user 消息不同。这意味着 API prompt cache 可以最大化命中。

### 借鉴价值
> 如果 OpenClaw 实现 subagent 上下文继承，可以参考这个优化：
> - 只传必要消息（task description + 关键上下文）
> - 避免每次都传完整 system prompt

## 六、Confidence Rating（预测置信度）

Claude Code 有一个 `confidenceRating` 系统：
- 当 AI 做出预测/判断时，记录置信度
- 下次遇到类似场景，对比置信度调整行为

### 借鉴价值
> 低 —— 需要长期追踪数据，当前数据量不够支撑

---

## 七、最值得借鉴的 3 个设计

### 🥇 第一名：Memory Taxonomy + WHAT NOT TO SAVE 规则
**可直接落地**：整理 MEMORY.md，删除那些"可从项目推导"的内容，聚焦用户偏好和独特规则。

### 🥈 第二名：ExtractMemories 思路
**可以工具化**：做一个 `memory-extractor.sh`，在会话压缩时运行，AI 判断哪些内容值得升权。

### 🥉 第三名：Drift Detection
**可增强 memory-tier.py**：加一个"记忆新鲜度检查"——定期验证 MEMORY.md 里的项目 facts 是否还正确。

## 八、不值得借鉴的模块

| 模块 | 原因 |
|------|------|
| Confidence Rating | 数据量不够，ROI 低 |
| Agent Summarization | 需要 fork 支持，OpenClaw 没有 |
| In-process Teammate | 需要进程内通信机制，复杂 |
