# Kimi Code 斜杠命令手册

> 维护日期：2026-06-05
> 适用基线：本机当前 **Kimi Code 0.9.0**
> 说明：本文重点整理 Kimi Code TUI 内的斜杠命令（slash commands），并与终端外的 CLI 子命令做边界区分。内容综合本机实测帮助、官方文档检索结果、既有工作流经验整理而成。

---

## 1. 先搞清楚：什么是“斜杠命令”

斜杠命令是你**已经进入 Kimi Code TUI** 后，在会话内输入的命令。格式通常是：

```text
/help
/login
/model
/usage
```

它们的作用不是像 shell 那样执行系统命令，而是：

- 控制 Kimi Code 自己的会话状态
- 登录 / 登出
- 查询当前会话信息
- 切换模型、模式、权限
- 管理历史会话
- 触发一些 Agent 内部能力

### 与 CLI 子命令的区别

| 类型 | 使用位置 | 例子 | 用途 |
|---|---|---|---|
| **斜杠命令** | 已进入 `kimi` TUI 之后 | `/login` | 管理当前 TUI 会话内能力 |
| **CLI 子命令** | 终端外直接调用 | `kimi login` | 在 TUI 外触发正式 CLI 功能 |

这个区别非常重要：

- `/login` ≠ `kimi login`
- `/provider` ≠ `kimi provider`
- `/sessions` ≠ `kimi export`

它们有时功能相近，但层级不同。

## 3. 登录与凭据相关命令

### `/login`

**作用：** 在 TUI 内登录或重新建立当前账号/平台凭据。

已知认知：

- 部分文档中 `/login` 别名为 `/setup`
- 可用于：
  - 初次登录
  - 登录态失效后的重建
  - 账号或平台切换时重新认证

**适用场景：**

- 你已经在 `kimi` TUI 里
- 想不退出会话就完成认证
- 某些能力因凭据过期而失效

**与 `kimi login` 的区别：**

| 命令 | 位置 | 更适合 |
|---|---|---|
| `/login` | TUI 内 | 会话中途补登录 |
| `kimi login` | 终端外 | 正式、直接、稳定地执行登录流程 |

### 实务建议

在脚本和自动化场景里，**优先认 `kimi login` 更稳**；
在人工使用 TUI 时，`/login` 更顺手。

---

### `/logout`

**作用：** 清除当前所选账号的登录凭据 / 让当前登录态退出。

**适合场景：**

- 想主动退出
- 想强制重新登录
- 需要切换账号
- 怀疑当前凭据状态异常

**注意：**

- 如果后续还要查 usage、拉取能力、使用某些账号绑定功能，退出后需要重新登录
- 对脚本类调用来说，登出后可能影响 `tools/kimi-usage.py` 等本地工具的正常取 token

---

### `/settings` / `/config`

这两个通常用于打开或管理 Kimi Code 的设置面板 / 配置界面。

**主要用途：**

- 调整使用偏好
- 查看当前设置
- 管理部分行为开关

在版本演进中，配置系统可能越来越强，因此：

- `/settings`
- `/config`

后续应持续关注是否出现分工变化。

---

### `/model`

**作用：** 切换当前会话所使用的模型。

**适用场景：**

- 想从默认模型切到其他模型
- 某任务需要不同风格 / 不同能力 / 不同成本
- 试验 provider 体系下不同模型表现

**与 CLI `-m` 的区别：**

| 方式 | 使用时机 |
|---|---|
| `/model` | 已进入 TUI，中途切换当前会话模型 |
| `kimi -m <model>` | 一开始启动单轮或新会话时直接指定 |

## 4. 帮助、状态与用量类命令

### `/help`

**作用：** 查看当前可用的 slash commands 和快捷帮助。

这是最基础、也是最建议首先尝试的命令。

**推荐用法：**

- 进入陌生版本的 Kimi Code 时先敲一次 `/help`
- 当你怀疑某个命令是否还存在、名字是否变了时，也先看 `/help`

---

### `/usage`

**作用：** 查看用量、配额、上下文占用或其他资源使用状态。

**适用场景：**

- 想看 token / quota 情况
- 想知道当前使用是否接近限制
- 想确认账号可用性

**与本地脚本 `tools/kimi-usage.py` 的关系：**

- `/usage`：TUI 内查看
- `tools/kimi-usage.py`：工作区脚本化查询

两者是互补关系：

| 方式 | 更适合 |
|---|---|
| `/usage` | 交互时顺手看一下 |
| `tools/kimi-usage.py` | 定时任务、飞书播报、脚本链路 |

---

### `/status`

**作用：** 查看当前会话或运行时状态。

**可能关注的信息：**

- 当前模式
- 当前会话上下文状态
- 当前模型 / 权限 / 会话运行情况

**适合场景：**

- 多轮任务进行到一半，想确认当前状态
- 模式切换后想核验是否生效
- 排查“为什么表现不对劲”时先做快速状态确认

---

## 5. 会话管理类命令

### `/new`

**作用：** 开启一个全新会话。

**适用场景：**

- 当前上下文太脏
- 想把新任务与旧任务完全隔离
- 想降低上下文污染和 token 消耗

这条命令对长时间使用 TUI 很重要，因为它直接关系到：

- 上下文纯净度
- 输出稳定性
- token 成本

---

### `/sessions`

**作用：** 浏览历史会话并恢复 / 切换。

**适用场景：**

- 想找回旧对话
- 想继续昨天的任务
- 想从多个历史任务中切换

**与 CLI 会话参数的关系：**

| 方式 | 场景 |
|---|---|
| `/sessions` | 已在 TUI 里，用交互方式选历史会话 |
| `kimi -S [id]` | 终端外直接指定会话恢复 |
| `kimi -C` | 继续当前工作目录上一轮会话 |

---

### `/undo`

**作用：** 撤回上一条提示词或最近一次输入。

**适用场景：**

- 刚输错了
- 上一轮提问方向错了
- 想回退一步重新组织任务

**价值：**

这是一个非常实用的“减轻误操作成本”命令。对于长对话尤其重要。

---

## 6. 模式 / 权限类命令

### `/yolo`

**作用：** 切换到更激进的自动批准模式。

简单理解：让 Kimi 在执行工具 / 操作时更少打断你确认。

**适合场景：**

- 任务范围清楚
- 对当前工作区和操作有把握
- 希望减少审批打断

**风险：**

- 自动化程度越高，误操作风险越高
- 在外部系统 / 重要文件 / 高价值环境下要谨慎

**与 CLI `--yolo` 的区别：**

| 方式 | 场景 |
|---|---|
| `/yolo` | 进入会话后动态切换 |
| `kimi --yolo` | 启动时就进入该模式 |

---

### `/plan`

**作用：** 切换或进入计划模式。

一般用于：

- 先规划再执行
- 拆步骤
- 写方案 / 列出行动路径

**适合场景：**

- 复杂任务
- 多阶段任务
- 想先看计划再动手

**与 CLI `--plan` 的区别：**

| 方式 | 场景 |
|---|---|
| `/plan` | 会话中途切换进入计划态 |
| `kimi --plan` | 启动即进入计划态 |

---

### `/btw`

**作用：** 发起“旁路对话”或侧通道提问，不打断主任务主线。

这是 0.9.0 非常值得关注的新能力之一。

**可以把它理解成：**

- 主线程继续干正事
- 你插入一个“顺嘴问一下”的问题
- 尽量不污染主任务上下文节奏

**适合场景：**

- 主任务执行中临时想确认一个小问题
- 想插播说明、补充、旁问
- 不希望打断主任务推进

**价值：**

这类命令对长任务尤其有意义，因为它直接帮助：

- 降低上下文污染
- 保持任务主线稳定
- 让多意图对话更自然

---

### `/goal`

**作用：** 设置持续推进的目标模式。

`/goal` 是 0.8.0 之后值得重点关注的能力，后续还演进到了“目标队列”概念。

**适合场景：**

- 你不是只想问一轮
- 而是希望 Agent 围绕一个目标持续推进
- 更像“交任务”而不是“问问题”

**价值：**

它让 Kimi Code 更像一个持续工作的 Agent，而不是一次性问答工具。

**前瞻说明：**

后续官方文档中已经出现 Goal Queue 方向，说明 `/goal` 很可能会继续增强，而不只是一个简单模式切换。

---

## 7. 迁移期需要谨慎理解的命令

### `/connect`

这是一个很容易误解的历史命令。

旧资料里它常被描述为：

- 从模型目录中选择并配置 provider / model

但从版本演进看，应注意：

- provider 管理能力正在增强
- `kimi provider` 已成为顶层 CLI 子命令
- `/connect` 的核心定位可能正在被 provider 体系逐步吸收或替代

**因此不建议把 `/connect` 再当成“最核心的稳定命令”来教。**

更稳的写法应是：

> `/connect` 属于历史文档中出现过、与 provider / model 配置相关的命令；但在较新版本里，需要结合 provider 体系一起理解，不应孤立地当成唯一入口。

---

### `/provider`

本次资料里，provider 更明确地以 CLI 顶层子命令 `kimi provider` 的方式出现。

因此：

- `/provider` 是否在 TUI 内仍作为主要入口
- 它与 `kimi provider` 的关系是否完全一致

这两点都要带着“版本迁移视角”看，不宜武断。

当前更稳的知识表达方式是：

- **TUI 层面可能存在 provider 相关入口**
- **CLI 层面明确存在 `kimi provider`**
- **provider 体系正在替代更老的连接/配置思路**

## 8. 斜杠命令 vs CLI 子命令：一张对照表

| 需求 | TUI 内斜杠命令 | 终端外 CLI 子命令 / 参数 |
|---|---|---|
| 登录 | `/login` | `kimi login` |
| 登出 | `/logout` | （如支持）`kimi logout` 或重新登录处理 |
| 切模型 | `/model` | `kimi -m <model>` |
| 看用量 | `/usage` | `tools/kimi-usage.py` / 其他脚本化方式 |
| 看状态 | `/status` | `kimi --help` / 运行态观察 |
| 新会话 | `/new` | 重新启动 `kimi` |
| 列历史会话 | `/sessions` | `kimi -S [id]` / `kimi -C` |
| 撤回上一条 | `/undo` | 无直接等价物 |
| 切 Yolo | `/yolo` | `kimi --yolo` |
| 切 Plan | `/plan` | `kimi --plan` |
| 设置目标 | `/goal` | 无完全等价顶层参数 |
| provider 管理 | `/provider`（需结合版本理解） | `kimi provider` |

## 9. 工作流建议

### 场景 A：我只是想问个问题

推荐：

```bash
kimi -p "你的问题"
```

不一定要进入 TUI，更不需要 slash commands。

---

### 场景 B：我要做多轮复杂任务

推荐：

```bash
kimi
```

然后按需要使用：

- `/plan`
- `/goal`
- `/usage`
- `/status`
- `/sessions`

---

### 场景 C：我在执行任务中途发现登录态有问题

- 人工交互时：优先试 `/login`
- 脚本/自动化时：优先走 `kimi login`

---

### 场景 D：我怕上下文越来越脏

优先考虑：

- `/new`
- `/undo`
- `/sessions`
- `/btw`

这些命令本质上都在帮助你管理上下文质量。

---

## 10. 当前最值得记住的几条结论

1. **`/login` 和 `kimi login` 不是一回事**
   - 前者是 TUI 内命令
   - 后者是官方 CLI 子命令

2. **`/usage` 和 `tools/kimi-usage.py` 是互补关系**
   - 一个是交互态顺手查
   - 一个是脚本化定时查询

3. **`/btw` 是 0.9.0 非常值得关注的新能力**
   - 适合旁路提问，不打断主任务

4. **`/goal` 代表 Kimi Code 正在往“持续推进型 Agent”方向走**
   - 不只是普通聊天

5. **`/connect` 需要带历史迁移视角看**
   - 不要再把它当唯一稳定入口

6. **slash commands 与 CLI 子命令必须分层理解**
   - 否则很容易把 TUI 内能力和终端外能力混淆

---

## 11. 后续维护建议

这份文档后续应按下面方式维护：

### 当官方版本更新时

优先关注：

- `/btw` 是否扩展
- `/goal` 是否进一步变成完整目标队列体系
- `/provider` / `/connect` 的边界是否继续变化
- `/login`、`/settings`、`/usage` 是否有别名或行为调整

### 当本机 CLI 再升级时

至少要补一次：

```bash
kimi --help
```

并在 TUI 内重新确认：

```text
/help
```

确保文档与实物一致。
