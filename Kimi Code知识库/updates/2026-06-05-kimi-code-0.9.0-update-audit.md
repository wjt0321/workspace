## 1. 本机版本与升级结果

### 当前状态

- 升级前本机 `kimi --version`：`0.9.0`
- 执行 `kimi upgrade` 的结果：提示有新版本 `0.10.1`，但当前安装源被判定为 `unsupported package manager or layout`
- 初次核验时，因 registry 仍指向 `https://registry.npmmirror.com`，`npm view @moonshot-ai/kimi-code versions --json` 与 `dist-tags` 只返回到 `0.9.0`
- 进一步切换到 npm 官方源 `https://registry.npmjs.org/` 复查后，已可确认：
  - `versions` 包含 `0.10.0`、`0.10.1`
  - `dist-tags.latest = 0.10.1`
  - `@moonshot-ai/kimi-code@0.10.1` 的 tarball 可正常解析
- 说明：**`0.10.1` 确实已经发布到 npm，之前“npm 尚未发布”的判断属于镜像同步滞后 / 源差异导致的时序性误判**

### 实际处理

已执行：

```bash
npm config set registry https://registry.npmjs.org/
npm install -g @moonshot-ai/kimi-code@0.10.1
kimi --version
```

处理中还确认到：

- 首次安装 `0.10.1` 时不再报 `ETARGET`，而是报 Windows `EBUSY`
- 被占用文件为本地原生模块：`clipboard.win32-x64-msvc.node`
- 用 `tasklist | findstr /I "kimi node"` 检查后，发现大量 `Kimi.exe`、`kimi-webbridge.exe` 与多个 `node.exe` 仍在运行
- 首次清理命令因把多个 `taskkill` 与安装步骤用 `&&` 串联，而某一步“未找到进程”导致整条命令链短路，安装实际未执行
- 改为分步处理、结束残留 `node.exe` 后，再次执行安装成功

结果：

- 安装成功
- 当前版本已升级为：`0.10.1`

### 结论

- **本机现已成功升级到 `0.10.1`**
- 真正阻塞升级的不是“包不存在”，而是：
  1. 初次判断时误用了镜像源结果
  2. Windows 下运行中的 `Kimi.exe` / `node.exe` 占用了全局安装目录内的 native 模块，导致 `EBUSY`
- 后续凡遇到“看得见新版本但装不上”的情况，应先区分：
  - registry / 镜像可见性问题
  - 本地文件锁 / 进程占用问题

## 3. 近期版本亮点（按本次官方检索整理）

> 注意：以下内容以本次 Kimi 官方文档搜索结果为主；其中 `v0.10.x` 因 npm 尚未发包，先作为“上游文档已提及 / 本机暂未可装”的前瞻信息保留。

### v0.10.x（前瞻信息，npm 暂不可装）

- **目标队列（Goal Queue）**：支持为 Agent 预先排队多个目标，逐个推进
- **`update-config` 内置 Skill**：允许 Agent 修改自己的配置
- **实验功能持久化面板**：对实验性功能做更明确的启停管理
- **`kimi doctor`**：新增配置体检命令，用于检查 `config.toml` / `tui.toml`
- **后台自动更新**：检测到新版本后可自动处理更新（文档层面已提及）

### v0.9.0（本机当前版本）

- **ACP 支持**：新增 `kimi acp`，可作为 Agent Client Protocol 服务端接入 IDE / 外部系统
- **`/btw` 旁路对话**：不打断主任务的侧通道提问
- **Windows 外部编辑器兼容修复**：减少 `/bin/sh` 依赖，Windows 体验更稳

### v0.8.0

- **Goal 模式**：引入跨轮次持续推进任务的目标模式
- **Provider 管理 CLI**：新增 `kimi provider` 子命令
- **后台结构化提问**：等待用户输入时可继续推进其他步骤
- **`/undo`**：撤回上一条提示
- **`kimi upgrade`**：CLI 内置升级入口

### v0.7.0

- **`/provider` 交互式供应商管理器**：逐步替代旧的 `/connect`
- **`KIMI_MODEL_ADAPTIVE_THINKING`**：支持更精细的思维模式控制

### 更早的关键变化

- `0.6.0`：模型切换环境变量、GitHub URL 插件安装、移除默认 1000 步上限
- `0.5.0`：定时任务（Cron）、`/auto` 权限模式、审批 diff 展示

## 6. 旧版 `kimi-cli` / `.kimi` 数据审计：还有没有用？

本机检查到：

- 存在旧目录：`C:\Users\wxb\.kimi\`
- 同时存在新目录：`C:\Users\wxb\.kimi-code\`
- `C:\Users\wxb\.kimi\.migrated-to-kimi-code` 已存在，说明**迁移动作已经做过**

### 旧目录里仍存在的主要内容

- `config.toml`
- `credentials/`
  - `kimi-code.json`
  - `kimi-code.lock`
- `logs/`
- `sessions/`
- `plugins/`
  - 如 `plugins/kimi-datasource/`
- `plans/`
- `telemetry/`
- `user-history/`
- `device_id`
- `kimi.json`
- `kimi-claw/`

### 我的判断：哪些还有价值

#### 仍有价值，建议保留 / 审阅后再决定

1. **`sessions/`**
   - 可能含历史会话与工作轨迹
   - 如果教主还想追溯旧任务思路，这部分有价值

2. **`plugins/`**
   - 自定义插件尤其是 `kimi-datasource` 可能仍有复用价值
   - 应核对新版本是否仍兼容、是否已被更现代的 skill / provider 体系替代

3. **`logs/`**
   - 对排查迁移前行为、登录异常、兼容问题有用
   - 不是长期高价值资料，但短中期可留

4. **`config.toml` / `kimi.json` / `device_id`**
   - 对理解旧版配置结构、迁移痕迹有价值
   - 不建议立即删

#### 价值较低，可后续考虑归档或清理

1. **`telemetry/`**
   - 一般分析价值低
2. **`user-history/`**
   - 若只是输入历史，长期价值有限
3. **重复凭据文件**
   - 若新目录凭据已稳定，旧目录凭据更多是兼容保底

### 结论

- **旧版数据并非“完全没用”**
- 它更像是：
  - 一部分是迁移遗产
  - 一部分是历史可追溯资产
  - 一部分是低价值运行残留

### 建议策略

不要立刻删旧目录，而是分三层处理：

1. **保留**：`sessions/`、`plugins/`、关键配置
2. **观察**：`logs/`
3. **后续可归档/清理**：`telemetry/`、`user-history/`、明显重复的缓存性文件
