# 06 - 自动上下文压缩系统（Auto-Compact）

## 核心文件

- `src/openharness/services/compact.py` — 压缩逻辑（完整实现）
- `src/openharness/services/token_estimation.py` — token 估算

> **重要：** OpenHarness 的 compact 系统是直接从 Claude Code 翻译过来的（文件头部有注明），所以跟 Claude Code 的 compaction 几乎完全一致。

---

## 1. 两层压缩架构

```
触发阈值（token 估算）
        │
        ▼
┌───────────────────────┐
│   Microcompact（廉价）  │  ← 第一道防线，不调 LLM
│  清除旧工具结果内容      │
└───────────────────────┘
        │
   重新估算 token
        │
        ▼
┌───────────────────────┐
│   Full Compact（昂贵）  │  ← 调 LLM 生成摘要
│  LLM summarization     │
└───────────────────────┘
```

## 2. Microcompact

### 可压缩的工具类型

```python
COMPACTABLE_TOOLS = frozenset({
    "read_file", "bash", "grep", "glob",
    "web_search", "web_fetch",
    "edit_file", "write_file",
})
```

> 注意：不含 `mcp__*` 工具，不含 `task_*` 工具，不含 `agent`。

### 清除策略

```python
def microcompact_messages(messages, keep_recent=5):
    """
    1. 收集所有 COMPACTABLE_TOOLS 的 tool_use_id
    2. 保留最近 keep_recent 个的结果
    3. 其余工具结果 → 替换为 "[Old tool result content cleared]"
    """
```

**特点：**
- 不调 LLM，极快
- 只清内容，保留工具调用结构（`ToolUseBlock` + `ToolResultBlock` 都在）
- `keep_recent=5` 保证最近的工作上下文不丢失
- 返回 `tokens_saved`，用于判断 Full Compact 是否还需要

### 清除效果示意

```
# 之前
ToolUseBlock(id="t1", name="read_file", input={...})
ToolResultBlock(tool_use_id="t1", content="...500行代码...")  ← 被清除

ToolUseBlock(id="t2", name="bash", input={...})
ToolResultBlock(tool_use_id="t2", content="...build输出...")  ← 保留

# 之后
ToolResultBlock(tool_use_id="t1", content="[Old tool result content cleared]")
ToolResultBlock(tool_use_id="t2", content="...build输出...")  ← 保留
```

## 3. Full Compact（LLM Summarization）

### 触发时机

```python
AUTOCOMPACT_BUFFER_TOKENS = 13_000
MAX_OUTPUT_TOKENS_FOR_SUMMARY = 20_000

def get_autocompact_threshold(model):
    context_window = get_context_window(model)  # 默认 200k
    reserved = min(MAX_OUTPUT_TOKENS_FOR_SUMMARY, 20_000)
    return context_window - reserved - AUTOCOMPACT_BUFFER_TOKENS
    # = 200000 - 20000 - 13000 = 167000（对于 200k 上下文模型）
```

### Compact Prompt 结构（精华）

**Preamble（强制单文本输出）：**
```
CRITICAL: Respond with TEXT ONLY. Do NOT call any tools.
- Do NOT use read_file, bash, grep, glob, edit_file, write_file, or ANY other tool.
- You already have all the context you need in the conversation above.
```

**Base Prompt（要求生成结构化摘要）：**
```
Your task is to create a detailed summary of the conversation so far.
...

<analysis>:
- Every user request and intent
- The approach taken and technical decisions made
- Specific code, files, configurations discussed
- All errors encountered and how they were fixed

<summary>:
1. **Primary Request and Intent**
2. **Key Technical Concepts**
3. **Files and Code Sections** (with paths and line numbers)
4. **Errors and Fixes**
5. **Problem Solving**
6. **All User Messages** (preserve exact wording)
7. **Pending Tasks**
8. **Current Work**
9. **Optional Next Step**
```

**Trailer：**
```
REMINDER: Do NOT call any tools. Respond with plain text only.
```

### 摘要注入消息

```python
def build_compact_summary_message(summary):
    return (
        "This session is being continued from a previous conversation "
        "that ran out of context. The summary below covers the earlier "
        "portion of the conversation.\n\n"
        f"{formatted_summary}"
        "\n\nRecent messages are preserved verbatim."
    )
```

## 4. AutoCompactState 状态机

```python
@dataclass
class AutoCompactState:
    compacted: bool = False           # 是否压缩过
    turn_counter: int = 0             # 循环次数
    consecutive_failures: int = 0     # 连续失败计数
```

**失败退避：**
- 连续失败 3 次 → 停止自动压缩（`MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES = 3`）
- 防止在 API 有问题时无限重试

## 5. 与 OpenClaw truncate.py 对比

| 方面 | OpenHarness | OpenClaw |
|------|------------|---------|
| 压缩时机 | 每次 `run_query` 循环前检查 | Cron Job（每天 01:00） |
| 压缩策略 | Microcompact + LLM summarization | 语义截断（保留 code/question/conclusion） |
| 保留内容 | 最近 5 个工具结果 + LLM 摘要 | 按语义块优先级保留 |
| Token 计算 | `estimate_tokens` + 4/3 padding | 字符数估算 |
| 失败处理 | 3 次连续失败后停止 | 无对应机制 |
| 摘要生成 | LLM 完整摘要 | 无（只是截断） |

## 6. 关键启发

1. **两层压缩**设计很妙：先用免费的 Microcompact 探路，如果够了就不用调 LLM
2. **Microcompact 保留工具结构**很重要——这样对话依然知道执行了什么工具，只是丢了输出内容
3. **Prompt 设计**的强制单文本输出（禁用 tools）避免 Compact 过程本身再触发工具
4. **连续失败退避**是防雪崩的好设计
5. **Claude Code 源码引用**说明 OpenHarness 是认真研究过竞品的
